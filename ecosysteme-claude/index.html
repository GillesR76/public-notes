Architecture en trois couches — Chat & Cowork

Dans les interfaces Chat et Cowork, trois couches s'emboîtent. Cette architecture est propre à ces deux environnements — Claude Code a son propre système (voir section 07).

Plugin (bundle)
Skill (atomique)
Command (prompt réutilisable ± orchestrateur)
02

Commands — définition précise

Un command est un fichier .md stocké dans commands/ qui encapsule une tâche nommée, déclenchable via /nom. Son contenu est du langage naturel décrivant la tâche à accomplir — il peut être aussi simple qu'un paragraphe, ou aussi structuré qu'une séquence ordonnée invoquant plusieurs skills. Le slash command est le mécanisme de déclenchement ; l'orchestration de skills est un usage avancé, non une propriété intrinsèque.

PropriétéValeur
FormatFichier .md — contenu en langage naturel / markdown
DéclenchementSlash command explicite : /nom-du-command
Contenu minimalPrompt décrivant une tâche — sans référence à des skills
Contenu avancéPrompt qui invoque explicitement skill A, puis skill B, puis skill C en séquence
Localisation Chat/Coworkcommands/ dans le dossier plugin
Localisation Claude Code.claude/commands/nom.md — voir section 07
Disponibilité sans pluginChat/Cowork : non (doit être dans un plugin). Claude Code : oui (dossier local).
Distinction clé : un command simple (/summarize) peut être un unique paragraphe de prompt. Un command orchestrateur (/linkedin-post) sera un fichier structuré listant les skills à charger et l'ordre d'exécution. Les deux formats coexistent.
03

Déclenchement — auto vs explicite (Chat & Cowork)

Le déclenchement d'une skill repose sur la correspondance entre la requête de l'utilisateur et la description dans le frontmatter SKILL.md. Le déclenchement d'un command est toujours explicite.

04

CLAUDE.md vs instructions de projet

Deux mécanismes distincts selon l'environnement. Confondre les deux génère des attentes incorrectes sur la persistance et la portabilité de la configuration.

CritèreCLAUDE.md (Claude Code)Instructions de projet (Chat / Cowork)
LocalisationFichier .claude/CLAUDE.md ou racine du repoInterface UI — stockées côté Anthropic, liées au projet Chat
Versionné dans gitOui — fait partie du repo, partageable avec l'équipeNon — attaché à l'utilisateur/projet UI, non versionné
ChargementAutomatique à chaque session Claude Code dans ce répertoireInjecté automatiquement dans chaque conversation du projet Chat
Contenu typiqueCommandes essentielles, architecture, conventions, règles de travail, chemins d'accèsTon, style, contexte utilisateur, instructions globales de comportement
PortabilitéSuit le repo — disponible sur toute machine qui clone le projetLiée au compte — non portable par le repo
HiérarchieCLAUDE.md racine + CLAUDE.md sous-dossiers (scope local)Instructions globales (userPreferences) + instructions par projet
Référence à des skillsPossible — chemins locaux vers des fichiers .md d'instructionsSkills déclenchées via frontmatter, pas via le texte des instructions

Cas concret : le fichier CLAUDE.md doit se trouver à la racine du projet. Il encode les règles non négociables que l'on souhaite implémenter (conventions de code, architecture, workflow de review). Ces règles suivent le repo et s'appliquent automatiquement dans Claude Code. Elles sont distinctes des instructions de projet configurées dans l'UI Chat qui encodent le ton et le workflow de conversation.

CLAUDE.md — exemple de structure
# Structure CLAUDE.md
## Project Overview          → contexte général, versions
## Commandes essentielles    → commandes de build et test
## Architecture              → couches, modules, conventions
## Règles de travail         → workflow de review, CI, gestion des branches
## Discipline lecture        → conventions de lecture du code
05

Notion de "projet" selon l'environnement

Le terme "projet" recouvre des réalités différentes dans Chat/Cowork d'un côté, et Claude Code de l'autre.

Chat
Projet UI
Espace créé dans l'interface claude.ai. Regroupe : instructions persistantes, fichiers uploadés, historique des conversations. Accessible depuis n'importe quel appareil (lié au compte). Portée : conversations isolées dans cet espace.
Cowork
Projet UI (idem Chat)
Même notion de projet que Chat — espace avec instructions, fichiers, historique. Cowork y ajoute des capacités agentiques (navigation, desktop). Pas de notion de "projet répertoire" au sens développement.
Claude Code
Pas de projet UI
Aucune notion de projet dans une interface. Le "projet" est le répertoire de travail courant (cwd). La configuration passe par .claude/ dans ce répertoire. Pas d'historique de conversations persistant multi-sessions comme dans Chat.
Point d'attention : on ne peut pas créer un "projet Claude Code" depuis claude.ai. Claude Code est un outil CLI/IDE — il opère sur le filesystem local. La persistance de contexte entre sessions passe par CLAUDE.md et .session-state.md, pas par une interface de projet cloud.
06

Interfaces Claude Code — terminal vs IDE

Claude Code peut être utilisé de deux manières, avec des différences d'ergonomie significatives.

ModeTerminal (CLI)IDE — VS Code / JetBrains
AccèsCommande claude dans le shellExtension VS Code ou fenêtre contextuelle latérale
InteractionChat textuel dans le terminal — input/output séquentielPanneau latéral avec accès aux fichiers ouverts, diff inline, contexte de l'éditeur
Contexte fichierExplicite — il faut passer les fichiers en argument ou via promptAutomatique — fichier actif et sélection transmis au contexte
Diff / patchAffiché dans le terminal, appliqué via commandeDiff interactif dans l'éditeur — accepter/refuser par bloc
Slash commandsDisponibles — /nom dans le prompt terminalDisponibles — palette de commandes ou champ de saisie IDE
CLAUDE.mdChargé automatiquement depuis le cwdChargé automatiquement depuis le répertoire racine du workspace

Le choix entre les deux modes dépend de la nature de la tâche. Le mode VS Code est adapté aux modifications unitaires : lecture de fichier, diff ciblé, correction localisée. Le terminal convient aux tâches longues ou séquentielles : scripts, sessions autonomes, enchaînements de commandes. Les deux modes lisent le même CLAUDE.md et partagent la même configuration.

07

Skills, Commands & Extensions dans Claude Code

Claude Code n'utilise pas le système de plugins/skills de Chat/Cowork. Les mécanismes équivalents sont différents.

BesoinChat / CoworkClaude Code
Instructions persistantes domaineSkill avec SKILL.md + frontmatterSection dans CLAUDE.md ou fichier .md référencé depuis CLAUDE.md
Déclenchement auto par descriptionOui — via frontmatter description:Non — Claude Code ne parse pas les frontmatters pour matcher des requêtes
Slash commandscommands/nom.md dans le plugin.claude/commands/nom.md — accessible via /nom dans Claude Code
Contenu d'un commandPrompt .md — peut orchestrer des skillsPrompt .md — peut référencer des sections de CLAUDE.md ou des fichiers locaux
Connecteurs / APIs externesConnectors dans le plugin (MCP, browser MCP)MCP servers configurés dans .claude/settings.json ou ~/.claude/settings.json
Plugins UI (bundle)Oui — déployable, zippableNon — concept absent. Pas de marketplace de plugins pour Claude Code.
Skills Chat dans Claude CodeN/ANon disponibles — le filesystem /mnt/skills/ est propre à Chat/Cowork
Important : skills installées dans Chat/Cowork sont stockées dans un filesystem interne propre à ces environnements, non accessible depuis Claude Code. Pour répliquer le comportement d'une skill dans Claude Code, il faut copier son contenu dans une section de CLAUDE.md ou dans un fichier .md local explicitement référencé (le déclenchement automatique par description n'existe pas dans Claude Code).
MCP servers dans Claude Code : c'est l'équivalent fonctionnel des connectors de plugin. Déclarés dans .claude/settings.json (scope repo) ou ~/.claude/settings.json (scope global utilisateur). Permettent à Claude Code d'appeler des APIs externes, des bases de données, des services tiers — sans passer par l'UI de plugin.

09

Choisir le bon environnement selon la tâche

Environnement Rôle Signal de déclenchement
Chat Contrôle & traçabilité Toute tâche nécessitant validation avant application
Cowork Tâches autonomes volumineuses Génération multi-fichiers, audit sans validation granulaire
Claude Code Développement dans le repo Accès filesystem, CLAUDE.md, diffs interactifs IDE
Terminal Exécution réelle Tests, git, build — Claude propose, Terminal applique

Règle de routage : modification avec validation intermédiaire → Chat. Tâche autonome et volumineuse → Cowork. Développement repo → Claude Code. Exécution système → Terminal.

Les quatre environnements se complètent — ils ne se substituent pas.
